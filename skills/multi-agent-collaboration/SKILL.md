# 多 Agent 群聊协作调度

## 适用场景

- 用户在“多Agent群聊”中发布复杂任务，需要多个角色并行协作。
- 多个对话绑定同一项目目录，并分别承担前端、后端、数据库、测试、文档、架构等职责。
- 需要把总任务拆成明确子任务，并让不同 Agent 在各自对话上下文中执行。

## 协作原则

1. 先识别成员角色和项目目录，再拆分任务。
2. 每个子任务必须分配给最匹配的 Agent，避免同一任务重复执行。
3. 子任务说明要包含目标、边界、输入、输出和验收标准。
4. 成员回复后，调度中心要汇总结果、风险、未完成项和下一步。
5. 对跨角色依赖进行显式标记，例如“后端接口完成后前端联调”。
6. 如果用户表达“写完之后、完成后、然后、再、接着、交给、最后”等顺序关系，必须按串行步骤执行，后一位 Agent 的输入要包含前一位 Agent 的产物。
7. 如果用户列出多个相互独立的角色任务，默认作为同一步并行分配，最终统一汇总。

## 推荐分配规则

- 前端 Agent：Vue/React 页面、交互、样式、浏览器构建、用户体验。
- 后端 Agent：FastAPI/API、业务逻辑、任务队列、权限、服务集成。
- 数据库 Agent：SQLite/迁移、表结构、索引、数据兼容。
- 测试 Agent：构建、语法检查、接口验证、回归风险。
- 文档 Agent：README、使用说明、变更记录。

## 调度输出格式

```markdown
## 多Agent任务计划

### 成员
- 前端：对话 #A，项目目录 ...
- 后端：对话 #B，项目目录 ...

### 分配
- 第 1 步（并行/串行）：[角色] 子任务、验收标准、依赖
- 第 2 步（串行）：[角色] 基于上一轮产物继续处理

## 执行过程
- 已分配给哪个 Agent / 对话
- 当前步骤是否依赖上一轮产物
- 成员是否已返回结果

### 汇总
- 已完成
- 风险
- 下一步
```

## 串行与并行示例

- “让写诗 Agent 写一首诗，写完之后让鉴赏 Agent 点评”：第 1 步分配给写作/写诗 Agent；第 2 步分配给鉴赏/评论 Agent，并把第 1 步诗作作为上游产物传入。
- “前端改页面，后端改接口，测试验证”：三个任务没有显式依赖时可同一步并行分配。
- “后端完成接口后前端联调，最后测试回归”：后端 -> 前端 -> 测试三步串行。

## 失败处理

- 如果没有可用成员，先提示用户把普通对话加入多Agent群聊并设置角色。
- 如果某个角色缺失，把相关任务分配给最接近的成员，并在汇总中标记角色缺口。
- 如果成员执行失败，保留失败原因，并给出可重试的子任务描述。

## 来源说明

本技能基于已安全扫描通过的 `task-based-multiagent` 技能思想改写：使用任务领取、状态回写和结果汇总机制，但适配 Codebot 的对话式多 Agent 群聊。
